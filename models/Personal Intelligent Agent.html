Threat Model Name: Personal Intelligent Agent

Owner: tomcjones

Reviewer:

Contributors:

Description: This model is for a personal agent that is in the control of a human with full agency of action. So it would not include employees or dependant children, for example.

Assumptions: The human is the holder of some digital device that will, at a minimum, include a secure execution environment that will be responsible of secrets, like private keys, that the human should not release under any normal circumstances.
The human is presumed to have configured a default set of policies that can be changed for each context if time is taken to target specific providers.
This model does not consider a distributed secure execution environment where the human could have the same secrets shared by multiple, otherwise independent, devices.
The user will not need an identifier that is used for any purpose other than maintaining a context with a cloud agent.

External Dependencies:
1. The personal Intelligent software should be signed and sourced by a supplier trusted by the human user.
2. The Wallet, here could be multiple, otherwise independent, devices. an application that is trusted by the user to access secrets in the secure execution environment.
In this model the wallet controls write access to the local model.
This model does not address the user control of the wallet which is needed to set policy.
3. A language model will be provided as the local model, from time to time, which will reset any existing model with new data.
4. The Secure Execution device has hardware protection of its contents and execution environment.
While it is possible that a smart card could fill some of the functions of an SEE, this model will assume the SEE fully supports confidential computing with user policies.

67. Cross Site Request Forgery [State: Not Started] [Priority: High]

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting, The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification:<no mitigation provided>
